## Why these items are v2.1, not v2.0

Each was raised, designed, and explicitly deferred during rc.7 scope grilling. Common pattern: **the gap is real but only becomes visible after 3-6 months of accumulated knowledge-base usage**. RC users will not perceive the defect; shipping rc.7 without these does not break the four-chain macro closure.

When the knowledge base grows to 50-100+ canonical entries OR the project sees 3-6 months of continuous use, these items become priority.

## Item 1 — Maturity Progression Mechanism

**Status**: Deferred from rc.7 / Grill #1
**Symptom**: Schema defines `draft → endorsed → stable` maturity tiers, but only the **demotion** direction is wired (via `orphan_demote` lint in doctor). No mechanism promotes a useful entry from `draft` to `endorsed` to `stable`. README sells "3 maturity tiers" but operationally only 1.5 are real.

**Why this becomes visible later**: Demotion threshold is 14/30/90 days. A `draft` entry that's consumed weekly avoids demotion but never advances. Users won't notice until 3+ months in, when they ask "why is every entry still draft?"

**Design (locked during Grill #1)**: Path B — half-automatic.

1. **doctor lint #27 `maturity_promote_candidate`** (deterministic, report-only):
   - Trigger: `consumed_count >= N AND age_days >= M AND maturity == 'draft'`
   - Default thresholds: `N=5, M=30 days` (configurable per fabric-config.json: `maturity_promote_consume_threshold` / `maturity_promote_age_days`)
   - Same logic for `endorsed → stable`: `N=10, M=90 days`
   - Doctor `--lint` reports; **does not auto-promote**
   - Maintains doctor's role as "consistency repairer, not baseline promoter" (per `doctor.ts:903` comment)

2. **`fab_review` action `modify` accepts `maturity_level` field**:
   - User in review skill sees "candidate for promotion: KT-DEC-0042, consumed 7×, 45 days old" → can opt-in promote via `fab_review.modify` mutation
   - Schema patch: `ReviewModifyChangesSchema` adds optional `maturity: "draft" | "endorsed" | "stable"` field
   - Server emits `knowledge_promoted_maturity` event (distinct from `knowledge_promoted` which is pending→canonical)

**Input signal from rc.7**: T06's `proposed_reason` enum becomes a useful priority signal — entries proposed via `explicit-user-mark` or `decision-confirmation` are candidates for faster promotion paths than `dismissal-with-reason` or `wrong-turn-revert`.

**Workload estimate**: 1-1.5 days (lint logic + review modify schema + tests + docs).

## Item 2 — Hook-Injection Consumption Signal Sidecar

**Status**: Deferred from rc.7 / Grill #2
**Symptom**: After rc.7 T9 (kill degenerate mode), `knowledge_consumed` event reliably fires from `fab_get_knowledge_sections`. But the **active injection** path — SessionStart hook broad listing + PreToolUse hook narrow injection — emits knowledge to Agent stderr **without writing any consumption event**. An entry actively shown to the Agent every Edit can still be demoted as "orphan" if Agent reads from the hint without calling `fab_get_knowledge_sections`.

**Why this is rc.7-survivable**: T9 closed the biggest silent-bypass surface (MCP `fab_plan_context` degenerate mode). Hook-injection remainder has smaller blast radius because hooks emit *summary lines* not full content — Agent often needs to call `fab_get_knowledge_sections` anyway for substance. The gap is real but secondary.

**Design (locked during Grill #2)**: Path B — lightweight sidecar.

1. **`.fabric/.cache/consumption-counter.jsonl`** (sidecar, mirrors T05's session-digest pattern):
   - SessionStart hook writes one line per emitted broad ID: `{stable_id, session_id, ts, source: "session_start"}`
   - PreToolUse hook writes one line per emitted narrow ID: `{stable_id, session_id, ts, source: "pre_tool_use", path: "<edited_path>"}`
   - Append-only, JSONL. Cleanup: doctor `--lint` archives entries older than 30 days

2. **`doctor` fold into `last_consumed_at`**:
   - On every lint run, doctor reads the sidecar
   - For each stable_id, the *max* of (`last_consumed_at` from events.jsonl) and (max ts from sidecar **with weight 0.2×**) becomes the effective decay timestamp
   - 0.2× weight encodes "passive view is worth ~1/5 of an explicit MCP fetch" — prevents banner blindness from artificially keeping junk alive

3. **New lint check #29 `decay_signal_diverged`** (warning-only):
   - Triggers when an entry has high sidecar count but zero MCP fetches in 30+ days
   - Suggests "narrow_paths matches but Agent isn't consulting full content — consider clearer summary line"

**Workload estimate**: 1.5-2 days (hook writes + doctor fold + lint #29 + tests).

## Item 3 — Canonical-vs-Canonical Semantic Dup/Contradict Scan

**Status**: Deferred from rc.7 / Grill #3
**Symptom**: fabric-review skill runs semantic dup/contradict checks during **pending review** (LLM compares incoming pending against canonical via `fab_review action="search"`). Once pending is approved into canonical, no further checks. Over 3-6 months: (a) drift — two canonical entries about the same topic written months apart become semantic duplicates; (b) reversal — older `proven` decision contradicts newer `endorsed` decision, both stay active and Agent gets injected both, contradictory state worse than no knowledge.

**Why this is rc.7-survivable**: Only emerges with substantial canonical mass (>30 entries) AND time elapsed (>6 months). RC users don't have either.

**Design (locked during Grill #3)**: Path B — deterministic + LLM combined.

1. **doctor lint #28 `canonical_content_similarity`** (deterministic, report-only):
   - Pairwise Jaccard or token-cosine on `title + summary` across all canonical entries
   - Threshold default `0.65` (configurable: `canonical_similarity_threshold`)
   - Reports top-N (default 10) suspicious pairs
   - **Does not auto-merge** — LLM judgment required for actual semantic comparison

2. **fabric-review skill `mode: "health"` LLM follow-up**:
   - Skill mode `health` activates when (a) doctor `--lint` reports `canonical_content_similarity` pairs OR (b) user explicitly invokes `/fabric-review health`
   - For each pair, LLM compares full bodies and emits verdict: `duplicate` / `contradict` / `complementary` / `false_positive`
   - User chooses action via AskUserQuestion: `merge` / `supersede` (old → archive, new stays canonical) / `keep_both` / `false_positive_acknowledged`

3. **New event `knowledge_superseded`** + frontmatter `superseded_by: <stable_id>`:
   - `plan_context` filter excludes superseded entries from injection (never shown to Agent again)
   - Audit trail preserved in events.jsonl

**Workload estimate**: 2-3 days (lint #28 + review health mode + supersede schema + event + plan_context filter).
