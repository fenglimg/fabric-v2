# 从 AGENTS.md 到 Fabric：给 AI Coding Agent 补上工程知识闭环

AI Coding Agent 越来越强，在真实工程里，存在一个反复出现的问题：Agent 能写代码，能读文档，能执行任务，但它很难稳定地“记住”一个项目长期积累下来的工程经验。

每次新会话开始，它都像重新加入项目一样，需要重新理解目录结构、技术栈、架构约定、历史决策和踩坑经验。于是开始使用 AGENTS.md、CLAUDE.md 这类规则文件，把项目约定写下来，让 AI 在每次对话中先读取这些规则。

这确实是一个重要起点。在实践中逐渐意识到：AGENTS.md 解决了“AI 应该读什么”的问题，却没有完整解决“知识如何持续沉淀、审核、治理和复用”的问题。

Fabric 就是沿着这个问题继续往下走的一次工程实践。

它不是要替代 AGENTS.md，而是想补上 AGENTS.md 缺失的那部分闭环：让项目知识能在 Claude Code、Codex 这些 AI Coding Agent 之间持续流动，并在日常开发中被提取、审核、复用和治理。

## 一、从 AGENTS.md 开始：静态规则还不够

AGENTS.md 的价值很清晰。

它可以作为 AI Coding Agent 的项目级入口，告诉 Agent：

- 当前项目的硬性约束是什么；
- 技术栈、目录结构、命名规范是什么；
- 哪些文档需要按需读取；
- 哪些模块有额外的局部规则；
- 修改代码后需要同步更新哪些说明。

一个合理的 AGENTS.md 结构通常不会把所有内容都塞进去，而是采用两层设计：

```text
project-root/
├── AGENTS.md
├── docs/
│   ├── guides/
│   ├── api/
│   ├── components/
│   ├── conventions/
│   └── troubleshooting/
└── src/
    └── components/
        ├── ui-dialog/
        │   ├── AGENTS.md
        │   └── docs/
        └── data-table/
            └── AGENTS.md
```

根目录的 AGENTS.md 始终加载，负责放硬规则、关键约定和文档索引。更详细的说明放在 docs/ 中，由 Agent 根据任务按需读取。复杂模块也可以有自己的 AGENTS.md，形成一棵轻量的项目知识树。

这套设计比把所有内容塞进一个超长规则文件要好得多。它减少上下文浪费，也让文档更容易维护。

但它仍然有几个问题。

第一，AGENTS.md 主要是静态入口。它能告诉 Agent“现在有哪些规则”，但不会自动判断一次开发过程里出现了什么新经验。

第二，它依赖人工维护。代码改了、架构变了、踩坑了，如果没有人主动更新，文档很快就会过时。

第三，不同 AI 工具之间容易分叉。Claude Code 有 CLAUDE.md，Codex 有自己的 AGENTS.md 和 skill 目录。规则一旦分散，就会出现“这个工具知道，那个工具不知道”的问题。

第四，它缺少知识生命周期。一条经验是临时判断、单次踩坑，还是已经被多次验证的稳定规则？它过期了吗？被引用过吗？应该提升、降级，还是归档？这些都不是 AGENTS.md 自己能解决的。

所以开始把问题从“如何写好 AGENTS.md”推进到另一个层面：

能不能让 AGENTS.md 继续作为入口，同时为它补上一套工程知识闭环？

这就是 Fabric 的起点。

## 二、从腾讯文章得到的启发：Harness 不是目的，知识才是核心

后来看到腾讯技术工程的一篇文章，核心观点是：Harness Engineering 的重点不应该只是工作流编排，而应该是知识沉淀。

现在讨论 AI 工程化时，很多注意力会放在这些问题上：

- 要不要多 Agent 协同；
- 要不要做 16 阶段工作流；
- 要不要给 Agent 加状态机；
- 要不要做自动测试、自动 review、自动发布；
- 要不要搭一个更复杂的 harness。

这些都重要，但它们更像是“管道”。真正能长期产生复利的，是管道里流动的知识。

一次架构决策、一次线上事故、一次调试踩坑、一个团队约定、一个业务流程，如果没有沉淀下来，下次 Agent 还是会重新推导，甚至重新犯错。反过来，如果这些知识能被持续积累、审核和复用，后面的每一次 AI 开发都会站在前一次的基础上。

腾讯文章里有几个概念，我认为非常值得吸收。

第一是知识类型化。工程知识不是一团模糊的文档，它可以拆成不同类型：

- decision：为什么选择这个方案，而不是另一个方案；
- pitfall：哪里容易踩坑，为什么这个坑不明显；
- guideline：未来类似场景应该怎么做；
- model：系统、领域、模块的心智模型；
- process：有顺序要求的流程和操作步骤。

Fabric 当前也采用同样的五类知识，目录名使用复数形式，并在 schema 里以枚举严格约束（decisions / pitfalls / guidelines / models / processes，多一个少一个都通不过 census 校验）：

```text
knowledge/
├── decisions/
├── pitfalls/
├── guidelines/
├── models/
└── processes/
```

第二是成熟度。知识不是写下来就天然可信。Fabric 当前 schema 使用三种成熟度，并且只有这三种（早期的 stable / endorsed 别名已经清掉，回到唯一权威枚举）：

- draft：刚提取出来，仍然需要验证；
- verified：已经被验证，可以被正常参考；
- proven：经过更充分验证，具有较高可信度。

第三是生命周期。知识需要进入、审核、提升、降级和归档。否则知识库只会越来越大，最后变成另一个没人敢信的文档堆。

第四是按需消费。Agent 不应该在每次会话里一次性读取全部知识，而应该根据当前任务、当前文件路径、当前意图，先拿索引，再读取相关条目。

这些概念成为 Fabric 的基础。

但 Fabric 没有照搬腾讯文章里的所有设计。它做了很多简化和取舍。

比如，腾讯文章里有更重的多层知识存储模型。Fabric 起步时只保留了最朴素的双根结构（项目内 .fabric/ 放团队知识，个人目录 ~/.fabric/ 放个人偏好）。但随着真实使用，这个双根结构本身也演进了——现在的知识底座是“可挂载的多 store + 三轴 scope”，这一点在第三部分专门讲，因为它是 Fabric 至今最大的一次架构演进。

再比如，腾讯文章强调工作流阶段里的知识注入，而 Fabric 当前选择了 harness-agnostic 的路线。它不要求你必须使用某个 16 阶段工作流，也不绑定某个 IDE 或 Agent 框架，而是通过现代 AI 工具普遍支持的能力接入：

- MCP tools；
- Stop hook；
- SessionStart / PreToolUse / PostToolUse 类 hook；
- Skill 模板；
- CLI 命令。

也就是说，Fabric 吸收的是“知识闭环”的内核，剔除的是“重型工作流编排”的外壳。

还有一些腾讯文章里的设计，Fabric 走到今天已经实现了一部分，也还留了一部分在后续规划里。已经能做的：给一个 store 关联 git remote（fabric store create / bind 时带 --remote），再用 fabric sync 同步，团队知识本身就能以一个可共享的 git store 形态在多个 repo 之间分发——这其实已经落地了早期设想里的“独立 team-knowledge.git”和“跨 repo 团队知识分发”。还没做的，主要是更细的权限边界：admin / contributor / reader 这类三角色权限模型，以及更深的组织级 federation，目前仍属于后续方向。所以 Fabric 当前版本聚焦的是跨客户端、本地优先的知识闭环，团队协作里的权限治理刻意先留轻。

所以，Fabric 对腾讯文章的理解可以概括成一句话：

工作流不是目的，知识 sustainment 才是目的。Fabric 先把知识从“被写下”推进到“能被持续消费、审核和治理”。

## 三、Fabric 当前的产品形态：底座 + 四个面

Fabric 当前的定位是：

cross-client knowledge sustainment for AI coding agents——一个面向 AI Coding Agent 的跨客户端知识 sustainment 层。

它不替代 Claude Code、Codex CLI，也不强制你使用某个固定工作流。它提供的是一套围绕知识的基础设施，让不同 AI 客户端可以通过同一套协议读写项目知识。

可以把 Fabric 拆成“一个底座 + 四个面”：底座是知识本身的存储与作用域模型；四个面分别是 CLI（给人和脚本）、MCP（给 Agent 运行时）、Hooks（在关键时机自动提醒）、Skills（让 AI 做复杂判断）。

### 0. 知识底座：从双根到可挂载多 store + 三轴 scope

这是相对早期版本最大的一次演进，也是理解 Fabric 现状的关键。

早期是固定双根：项目里一个 .fabric/，家目录一个 ~/.fabric/。这个结构简单，但很快撞墙：团队知识没法在多个 repo 之间共享，个人知识和项目知识混在一起也不好治理。

现在的模型是这样的：

- 知识的真源是“store”，而不是某个固定目录。每个 store 在自己的 git 树根持有一个 store.json，里面有一个内生的、不可变的 UUID——store 换了远程地址、换了别名，身份也不变。
- store 被“挂载”到本机全局配置 ~/.fabric/fabric-global.json，再由具体仓库通过 fabric store bind 声明“我这个项目要用哪些 store”。没绑定，知识就不会被读到。
- 写入有明确路由：个人范围的知识永远落到当前激活的 personal store；非个人范围的知识落到项目通过 fabric store switch-write 指定的写入 store。
- 重要的一点：项目本地的 .fabric/knowledge/ 不再是运行时知识源，它只是一次性导入的输入。运行时真正被读写的，是挂载的 store。

在这个底座之上，一条知识“会不会浮现给 AI”由三个互相正交的轴决定（这是 Fabric 里反复强调的 KT-MOD-0001）：

- semantic_scope（受众）：team / project:<id> / personal。决定“谁能看到”。个人知识必须待在 personal store，这是隐私红线，schema 层强制。
- relevance_scope（时机）：broad / narrow。broad 常驻、会话开始就列出；narrow 只在你编辑到匹配路径的文件时才浮现。这个值不用手写，由是否填了 relevance_paths 自动推导。
- store（物理库）：知识具体驻扎在哪个挂载的库里。没绑定该库，前两轴再匹配也读不到。

这三轴有个容易混淆的地方：名字会撞——“team”既是 semantic_scope 的一个取值，也可能是某个 store 的别名。所以当你困惑“为什么这条知识没浮现”时，Fabric 提供了 fabric audit why-not-surfaced <id>，逐轴诊断到底是 store 没绑、scope 不匹配，还是 narrow 时机没到。

### 1. CLI：给人和脚本使用

CLI 是确定性的命令行入口，不依赖 LLM 判断。当前的命令面比早期宽不少，可以按用途分几组：

接入与卸载：

```bash
fabric install        # 标准接入：扫描项目，安装 hooks、skills、客户端配置
fabric install --yes  # 非交互
fabric install --dry-run  # 只看计划不写文件
fabric uninstall      # 对称反向卸载
```

知识库运维（多 store 架构的核心新增）：

```bash
fabric store list           # 列出已挂载的知识库
fabric store bind <id>      # 声明本项目要用的知识库
fabric store switch-write <alias> --scope <scope>  # 设置默认写入目标
fabric sync                 # 多 store 的 git 同步（pull --rebase + push）
fabric sync --continue      # 解决 rebase 冲突后继续
fabric sync --abort         # 放弃本次 rebase
```

体检与治理：

```bash
fabric doctor               # 知识库 / hook / schema 健康检查
fabric doctor --fix         # 确定性修复
fabric audit cite           # cite 覆盖率报告
fabric audit conflicts      # 知识冲突 lint
fabric audit retired        # 扫描指向已退役工具/字段的引用
fabric audit why-not-surfaced <id>   # 诊断知识为何没浮现
fabric audit metrics        # 指标仪表板
```

查看与配置：

```bash
fabric info                 # 项目状态
fabric info --global        # 全局身份（原 whoami）
fabric info --recall        # 召回引擎状态
fabric inspect              # 显示本次 SessionStart 实际注入了什么（字节级一致）
fabric config               # 交互式编辑 .fabric/fabric-config.json
```

相比早期文档，这里有几处明确的变化：早期的 serve 已经被剥离到独立的实验包，不再属于主 CLI；whoami / status / scope-explain 三个命令合并进了 info；原来挂在 doctor 上的一堆审计 flag 独立成了 audit 命令组；store / sync 是随多库架构新增的。CLI 适合做安装、检查、升级、CI 和本地维护。

### 2. MCP：给 Agent 使用

MCP 是 Fabric 的运行时协议层。Agent 不应该靠硬编码读 store 目录，而应该通过 MCP 获取和写入知识。

当前是五个工具：

```bash
fab_recall         # AI 直接调用：改文件前先召回相关知识
fab_propose        # 提议一条 pending 知识
fab_archive_scan   # 扫描会话历史，找可归档候选
fab_pending        # 只读浏览 / 搜索 pending + canonical 知识
fab_review         # 写：approve / reject / modify / defer
```

这里有一个值得专门讲的设计，叫 lean recall（精简召回）：

fab_recall(paths=[...]) 一次调用就返回相关知识的“描述 + 磁盘读取路径（read_path）”，它不通过 MCP 直接灌正文。需要某条正文时，Agent 自己对那个 read_path 做一次原生 Read。

为什么这么做？因为把正文 eager 地塞进每次召回响应，是一笔恒久的上下文税——大部分时候 Agent 只需要描述索引就能判断该不该用，真要看正文再读一次也很便宜。这一条对应代码里反复出现的 KT-GLD-0005。这也是相比早期“两步召回（plan_context → get_sections）”的简化：从两步缩成一步，描述和读取路径在一个统一列表里返回。

其实这个“索引常驻、正文按需 Read”的形状并不新鲜，它和 Claude Code 自己的 Memory 机制是同一个套路——MEMORY.md 是一份常驻索引（每条记忆一行），具体记忆是独立文件，相关时才用 Read 取回。Fabric 的召回是有意识地往这个方向收敛的：代码里直接把 fab_recall 的返回结构注释成 “Memory-style shape”——一份动态生成的描述索引，加上一个去 Read 取正文的文件路径，没有服务端打包正文、切预算、两步 token 那套仪式。换句话说，lean recall 不是凭空设计的，而是借了一个已经被验证好用的形状。

召回内部怎么排序，也值得说一句，因为它不是只做一种检索，而是把两路信号叠起来（hybrid retrieval）：先用 BM25 词法检索（带中文分词）按词面重合度打底，再叠加一路向量语义检索——用一个在 CPU 上跑的小型 dense embedding 模型（中文场景默认 fast-bge-small-zh）算余弦相似度，把那些“说的是同一件事、但用词不同”的条目也捞回来。向量这一路现在默认开启，但它是优雅降级的：embedding 包（fastembed）是可选依赖，装不上、被关掉、或运行出错时，召回会自动退回纯 BM25 + 时近 + 路径相关 + 重要度的文本排序，行为不变。模型首次使用会下载到 ~/.fabric/cache/embed（只拉模型权重，不上传任何知识内容），可以用 fabric info --recall --warm 预热。

工具的角色也分两类：fab_recall 是 Agent 在干活时自己内联调用的；fab_propose / fab_archive_scan / fab_pending / fab_review 是由 Skill 在生命周期合适的时刻驱动的，不是用户日常手点的。

### 3. Hooks：在关键时机提醒

Fabric 不希望完全依赖用户记得“该沉淀知识了”。所以它通过 hook 在关键节点提醒 Agent。当前安装模板覆盖 Claude Code 和 Codex CLI 两端（Cursor 已移除）。核心 hook 包括：

- knowledge-hint-broad.cjs：SessionStart 阶段触发，会话开始时列出 broad 知识索引与 scope 普查；
- knowledge-pretooluse.cjs：PreToolUse（Edit/Write/MultiEdit）阶段触发，编辑前给出路径相关的 narrow 知识提示，并侧记编辑计数；
- cite-policy-evict.cjs：PreToolUse 阶段触发，检查改文件前是否有相关 recall，没有就给一条软提醒（是 nudge，不是 gate）；
- post-tooluse-mutation.cjs：PostToolUse 阶段触发，纯观察——记录 file_mutated（文件被改）和 knowledge_body_read（知识正文被读）事件；
- fabric-hint.cjs：Stop 阶段触发，判断要不要提醒归档（archive）、审核（review）、或冷启动回灌；
- session-end-marker.cjs：SessionEnd 阶段写一条会话结束标记。

Stop hook 的重点是判断：

- 是否距离上次知识提议已超过配置阈值，推荐调用 fabric-archive；
- pending 队列是否过多或过旧，推荐 fabric-review；
- 项目知识节点过少且接入后已过一段时间，推荐冷启动回灌。

这里也能看到一个新增的闭环：post-tooluse-mutation 记下的 knowledge_body_read，让 Fabric 能离线重建“知识被浮现 → 被引用 → 被编辑”这条漏斗，而不是只能猜。

这个设计很重要：hook 只负责提醒和记账，真正的判断交给刚经历过上下文的 AI，并且不阻塞主流程。

### 4. Skills：给 AI 做复杂判断

Skill 是 Fabric 中最贴近用户日常使用的一层。它不是底层 API，而是一套让 AI 在会话中执行复杂判断的工作流模板。当前是四个：

```text
fabric-archive
fabric-review
fabric-store
fabric-sync
```

fabric-archive 用于从一次或多次会话中提取值得保留的知识。它会判断哪些内容真的值得归档，分类为五类之一，然后通过 fab_propose 写入 pending。它还吸收了原来独立的 fabric-import：在 source mode 下，它能从已有的 git log 和 docs 里冷启动回灌候选知识，帮老项目从 0 到 1 建库。

fabric-review 用于审核 pending 或 canonical knowledge。它会根据上下文推断 review mode，并通过 fab_review 执行 approve、reject、modify、defer。它还收纳了两个子流程：retire（语义淘汰陈旧/孤儿/被取代的条目，守住“先降级、先抢救，不硬删”的红线）和 relate（按需补 related 关联边，默认不主动建边）。

fabric-store 和 fabric-sync 是两个“薄路由层”：把用户的自然语言意图（创建/挂载/绑定/列出/切换写库、同步知识库）路由到对应的 fabric store / fabric sync CLI 命令。真正干活和把守安全门的是 CLI，skill 只负责指路；sync 里唯一需要 AI 介入的，是 rebase 冲突时帮你看清两侧、给出合并建议，最后由你拍板 --continue 还是 --abort。

### 5. 知识存储：仍然是可读的 Markdown

无论 store 怎么挂载、scope 怎么分轴，Fabric 的知识文件仍然是 Markdown，不是黑盒数据库。每个 store 下就是 knowledge/{decisions,pitfalls,guidelines,models,processes}/ 这样的目录树，frontmatter 里记 semantic_scope、relevance、maturity 这些坐标。

这意味着可以直接用 Git 管理、审查、比较和迁移这些知识。MCP、CLI 和 Skill 是更方便的入口，但底层仍然是可读、可版本化的文件。

Fabric 当前的整体分工可以概括为：

底座（store + 三轴 scope）负责知识到底存在哪、对谁可见、何时浮现；CLI 负责安装和治理；MCP 负责运行时知识读写；Hook 负责触发提醒与记账；Skill 负责让 AI 做需要判断的提取与审核。

## 四、日常怎么用：从安装到知识闭环

Fabric 的使用可以分成两类：首次接入和日常循环。

### 1. 首次接入

安装 CLI：

```bash
# 正式版
npm install -g @fenglimg/fabric-cli
# 体验版
npm install -g @fenglimg/fabric-cli@next
```

进入目标项目：

```bash
cd <your-project>
fabric install
```

如果希望非交互执行：

```bash
fabric install --yes
```

如果只想看安装计划，不写文件：

```bash
fabric install --dry-run
```

安装后，Fabric 会完成几件事：

- 创建或刷新 .fabric/ 目录与项目配置；
- 配置个人仓库和团队仓库；
  - 个人仓库用于存储个人知识，团队仓库用于存储团队知识。
- 安装 fabric-archive、fabric-review、fabric-store、fabric-sync 四个 Skills；
- 安装两端（Claude Code / Codex CLI）的 hook 脚本和对应配置；
- 写入或合并 AI client 所需配置；
- 同步 AGENTS.md / CLAUDE.md 中的 Fabric 指针。

检查状态：

```bash
fabric doctor
```

### 2. 日常开发循环

一次典型的 Fabric 使用闭环是这样的：

```text
安装并绑定 store
  ↓
AI 正常开发
  ↓
SessionStart / PreToolUse hook 提供知识提示
  ↓
Agent 改文件前用 fab_recall 拿回相关知识的描述 + 读取路径
  ↓
需要正文时对 read_path 做原生 Read（被 PostToolUse 记为 knowledge_body_read）
  ↓
会话结束时 Stop hook 检测 archive / review 信号
  ↓
调用 fabric-archive 提取候选知识，fab_propose 写入当前写库的 pending
  ↓
调用 fabric-review 审核
  ↓
approve 后晋升为正式知识，分配稳定 ID
  ↓
fabric doctor / fabric audit 持续治理
  ↓
下次任务自动复用
```

比如一次真实业务场景：

你在项目里修一个认证模块的 bug。Agent 在改 src/auth/** 前，PreToolUse hook 先给出路径相关的 narrow 知识提示。Agent 再用 fab_recall(paths=["src/auth/..."]) 拿回相关历史 decision、pitfall、guideline 的描述索引，发现有一条关于 token refresh 的坑，于是对它的 read_path 做一次 Read 看清细节，避免重复犯错。

这次修复过程中，你又发现一个新的非显然问题：某个边界条件在测试环境不会触发，但生产配置下会导致 refresh token 被提前失效。

会话结束时，Stop hook 发现这次会话可能有沉淀价值，于是提醒调用 fabric-archive。fabric-archive 把这条经验整理成一个 pending pitfall，写进当前写库：

```yaml
type: pitfalls
maturity: draft
summary: refresh token 在生产配置下会被提前失效
semantic_scope: team
relevance_paths:
  - src/auth/**
```

随后你运行 fabric-review，确认这条经验确实有价值，于是 approve。它被分配稳定 ID，进入正式知识库。

下次任何 Agent 再修改 src/auth/** 时，这条 pitfall 就可以被提前注入上下文。

这就是 Fabric 想形成的闭环：不是把知识写进文档就结束，而是让知识在后续任务中继续发挥作用。

### 3. 历史项目冷启动

对于已经存在很久、但没有 Fabric 知识库的项目，可以用 fabric-archive skill 的 source mode。它不是简单扫描文件名，而是让 AI 基于已有的 git log、docs 和提交历史提取候选知识。所有候选先进入 pending，之后仍然需要 review。

这避免了一个常见问题：冷启动时一次性导入大量未经确认的“伪知识”，最后污染知识库。

### 4. 知识治理

知识库必须能增长，也必须能清理。Fabric 把治理拆成了 doctor（健康与修复）和 audit（审计与遥测）两块：

```bash
fabric doctor                # 健康检查
fabric doctor --fix          # 确定性修复
fabric audit conflicts       # 知识冲突
fabric audit cite            # cite 覆盖率
fabric audit retired         # 指向已退役工具/字段的引用
```

它会检查：pending 是否积压、stable id 是否重复、index 是否漂移、scope 是否一致、陈旧知识是否需要降级或归档、hook 和 skill 是否正确安装、事件账本是否完整。

这一步决定了 Fabric 不是一个“只进不出”的知识收集器，而是一个持续维护知识健康度的系统。配合 fabric-review 里的 retire 子流程，淘汰陈旧知识时也守着“先降级、先抢救，不硬删”的红线。

## 五、一些取舍和后续规划

Fabric 当前仍然是一个刻意控制了边界的里程碑。

它没有做完整的重型工作流平台，也没有试图替代 Claude Code 或 Codex。它只做一件事：让 AI Coding Agent 的工程知识能够跨会话、跨客户端、跨时间持续存在。

当前已经比较清晰的能力包括：

- 跨客户端 MCP-first 知识层（Claude Code + Codex CLI 两端）；
- 可挂载的多 store 知识底座 + 三轴 scope（受众 / 时机 / 物理库）；
- 五类知识类型、三阶段成熟度；
- archive → review → promote → doctor/audit 治理的闭环；
- 底座 + 四个面（CLI、MCP、Hooks、Skills）的产品形态；
- lean recall、body-on-demand 等几条降低上下文成本的机制；
- BM25 词法 + 可选向量语义的混合召回（fastembed，默认开启、可优雅降级，中文场景用中文 embedding 模型）；
- 本地优先、文件可读、Git 可管理。

后续还可以继续往几个方向推进。

第一是团队协作的成熟度。跨 repo 共享的 git store 已经能用，未来可以围绕一个约定俗成的 team-knowledge.git，把多人协作时的 onboarding、冲突处理和审核流转做得更顺。

第二是权限边界。当前 review 模型比较轻，未来可以引入 admin、contributor、reader 这类角色，把团队知识库的写入和审核边界做清楚。

第三是检索质量的持续打磨。向量语义检索已经默认开启、并和 BM25 词法检索混合，但仍有可做的空间：更好的中文 embedding 模型、召回结果的 rerank、以及严格离线场景下的模型分发（目前 fastembed 首次使用需要联网拉一次模型权重，不是硬气隙）。这些都得在 minimal-install“重活必须可选、可降级”的底线内推进，而不是反过来把检索做成必装的重型基础设施。

## 结语

AGENTS.md 是一个很好的起点。它让 AI Coding Agent 能够读取项目规则，不再完全从零开始。

但真实工程里，仅有规则入口还不够。我们还需要知道：

- 新经验如何被提取；
- 候选知识如何被审核；
- 稳定知识如何被复用；
- 过时知识如何被降级；
- 不同 AI 客户端如何共享同一套项目记忆。

Fabric 想补上的正是这一段。

它把 AGENTS.md 从一个静态入口，连接到一个持续运转的知识闭环中：store 底座负责知识存在哪、对谁可见、何时浮现，CLI 负责安装和治理，MCP 负责 Agent 运行时访问，Hook 负责提醒与记账，Skill 负责判断。

从这个角度看，Fabric 并不是“又一个文档工具”，而是一次关于 AI 工程知识如何持续存在的实践。

或者说：

AGENTS.md 让 AI 读懂项目规则，Fabric 让项目经验在每一次 AI 开发之后继续留下来。
