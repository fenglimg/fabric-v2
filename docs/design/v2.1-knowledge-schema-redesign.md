# v2.1 Knowledge Schema Redesign

> Status: **Design locked (2026-05-25)** — implementation deferred to v2.1-rc.1+
> Authority: `memory/project_l0_l1_l2_redesign_v21.md` (grill-me session 共识)
> Tracking: `docs/roadmap/v2.1.md`

## 1. Problem Statement

`AgentsMetaNode.level` 字段使用 `L0|L1|L2` enum,该 enum 跨 4 个 MCP tool 表面 + frontmatter + service code,实践中出现以下结构性问题:

### 1.1 `layer` 一词在 5 处含义不同 (term pollution)

| 出现位置 | 含义 |
|---|---|
| `AgentsMetaNode.layer` (rc.30 TASK-004 已删) | L0/L1/L2 loading semantic |
| `fab_extract_knowledge.layer` 入参 | `"team" \| "personal"` storage location (写到哪) |
| `fab_plan_context.layer_filter` 入参 | `"team" \| "personal" \| "both"` filter |
| `fab_review.filters.layer` / `fab_review.modify.changes.layer` | 同 storage filter / layer flip target |
| frontmatter `knowledge_layer` | `"team" \| "personal"` storage |

LLM 客户端读 schema 时无法仅凭字段名分辨它在指 loading semantic 还是 storage location。

### 1.2 `level` enum 三档实际只用一档

本仓 (`.fabric/agents.meta.json`) 22 个 canonical entry **全部 L1**(零 L2 零 L0)。L0/L1/L2 三档设计的实际使用率 = 1/3,另两档历史上从未真正承载语义。

### 1.3 加载逻辑分散在 3 个正交轴上(activation.tier + level + relevance_scope 双轨)

| 轴 | 字段 | 来源 |
|---|---|---|
| A | `activation.tier`: `"always" \| "path" \| "description"` | v1.x 时代,server 路由策略 |
| B | `level`: L0/L1/L2 (with `node.layer` dual,已删) | rc.5 二步式 |
| C | `relevance_scope`: `"narrow" \| "broad"` + `relevance_paths[]` | rc.5+ entry 自声明 |

轴 A 的 `"path"` + scope_glob 与轴 C 的 `"narrow"` + relevance_paths 语义高度重叠,**v1→v2 双轨**未清算。

### 1.4 `personal-override` 语义难以表达

旧 L0 含义混了 "personal storage" + "highest precedence",`personal_selectable`(我个人的可选偏好) / `team_required-override`(团队某规则我个人覆盖)等组合表达不了 — 单字段 enum 永远无法 cross-product 表达两个正交维度。

## 2. Proposed Schema

### 2.1 AgentsMetaNode (internal)

```ts
type AgentsMetaNode = {
  stable_id: string;                    // KT-DEC-0001 / KP-DEC-0001 — 前缀编码 scope
  scope: "team" | "personal";            // 储存维度
  file: string;
  hash: string;
  content_ref?: string;
  description?: RuleDescription;

  // 来自 frontmatter — 单一 relevance 字段,合并 activation.tier + relevance_scope
  relevance:
    | { match: "path"; paths: string[] }   // path 命中 → 直接加载 body
    | { match: "always" };                  // 永远候选 → 进 description_index,LLM 看 stub 自己挑
};
```

### 2.2 加载规则 (一句话)

- **`match: "path"` 命中 paths → 直接 inject body 进 system prompt**
- **`match: "always"` → description 进 index,LLM 在 plan_context 阶段看 stub 自己挑**

### 2.3 Precedence rule (同 stable_id 冲突)

```
1. 同 stable_id 冲突: personal scope 永远胜 team scope
   (KP-DEC-0001 命中时 KT-DEC-0001 被屏蔽)
2. cross-check: stable_id 前缀 (KT-/KP-) + scope 字段 + 磁盘路径 三处必须一致
   (doctor lint enforce,任意不一致 = error)
```

### 2.4 砍掉的清单

- `level: "L0" | "L1" | "L2"` 字段 + enum 整个
- `activation.tier: "always" | "description" | "path"` (v1.x) → 合并进 `relevance.match`
- `relevance_scope: "narrow" | "broad"` (rc.5+) → 合并进 `relevance.match` (narrow→"path",broad→"always")
- L2 required vs L1 selectable 双档 ceremony — 单档 (path → 自动, always → LLM 挑)
- `precedence: tuple([L2, L1, L0])` deprecated wire 字段
- `node.layer === "L0"` personal-override 分支
- `.fabric/bootstrap/README.md` L0 anchor + `l0Content` field (v1.x 残留)
- `node.priority` passthrough 字段 (rc.30 TASK-003 暂留 fallback,v2.1 真删)

## 4. Disk Format & Reader 兼容

### 4.1 双 schema 并存策略

v2.1-rc.1 + rc.2 两个 rc 双写双读:
- **Writer**: 用新 schema (`scope` / `relevance`) 写新 entry
- **Reader**: 同时认旧 (`knowledge_layer` / `relevance_scope` / `relevance_paths`) 和新 frontmatter,parse 时 normalize 为内部新 shape

### 4.2 v2.1-rc.3 删旧

完成所有 disk 文件 migration 后(`fabric doctor --fix-knowledge` 自动 migrate),v2.1-rc.3 删旧字段 reader / 删 deprecation warning。

### 4.3 stable_id 前缀决定 scope

`KT-...` → team,`KP-...` → personal。frontmatter `scope` 字段与前缀必须 cross-check 一致(doctor enforce)。

## 5. Migration Plan

### 5.1 自动 migration (doctor)

```bash
fabric doctor --fix-knowledge --migrate-v21
```

- 读旧 frontmatter (`knowledge_layer` / `relevance_scope` / `relevance_paths`)
- 转为新 frontmatter (`knowledge_scope` / `relevance: {match, paths?}`)
- atomic write per file
- emit `knowledge_migrated_v21` event per file (新 ledger event type)

### 5.2 confirmation 模式

默认 `--migrate-v21` 全自动应用。`--migrate-v21 --confirm` 模式 per-entry 显示 diff 让用户接受/跳过(对接 `fabric-archive` skill 风格)。

## 7. Open Questions (留给 v2.1-rc.1 启动时决议)

1. **Scope 维度要不要加 "workspace"?** — 当前只有 team/personal,但跨 repo 共享场景可能要 workspace。建议留 enum 可扩展空间,但 v2.1-rc.1 只支持 team/personal。
2. **`relevance.match` 除了 `"path"` / `"always"` 要不要加 `"tag"` (按 tech_stack 或 onboard_slot 匹配)?** — 当前 `relevance_paths` 只支持 glob,未来按 tech_stack tag 选可能更精准。但 v2.1 redesign 不增功能,保持 1:1 替换,新 match 类型留 v2.2。
3. **`doctor --fix-knowledge --migrate-v21` 自动 migrate vs per-entry confirm?** — 平衡 UX / 安全。建议:default 自动(因 entry 数小,本仓 22 个),`--confirm` flag opt-in per-entry diff 模式。
4. **v2.1-rc.1 deprecation warning 在 log/CLI/MCP error 哪几处发?** — 三处都发:CLI startup banner + MCP response `warnings[]` + ledger event `legacy_schema_used`。统一让 dogfood 与第三方都能看到。

## 8. Why this redesign

- **数据驱动**: 本仓 22 entry 全 L1,3 档设计的另两档使用率 = 0,over-engineering 实锤
- **简化认知负担**: 写规则时不再有 "required vs selectable" 决策点 (memory/feedback_clean_slate 对齐)
- **解决 layer 词义污染**: `scope` 词专属 storage 一处,不再跨 5 处含义不同 (memory/project_l0_l1_l2_layer_system 根治)
- **API 表面统一**: 4 个 MCP tool 同一概念用同一字段名,LLM 客户端学习曲线降低
- **未来可扩展**: 正交两字段(scope + relevance/load)可独立加新值,不破现有(单字段 enum 加值会破排序假设)
