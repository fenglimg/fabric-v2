# Fabric Protocol (AGENTS.md 2.0) — 双视角严格评估

> 用户提出的新架构：织物协议，5 组件（递归嵌套 / Intent Ledger / 动态 MCP / Docs-as-Views / 人机治理）
> 评估工具：Gemini 3.1 Pro（架构批判）+ Claude（跨端落地），均已严格对照 Phase 2 能力矩阵
> 产出日期：2026-04-18

---

## 1. 综合结论（一句话）

> **原始 Fabric Protocol needs-revision**——设计方向正确，但 5/5 组件都在"无 Hook 客户端 + AGENTS.md 开放标准语义"两个维度存在缺陷。**砍掉'强制性'宣称、改用 git 层替代客户端层、元数据降级为 HTML 注释**后，可以落地为一个真正跨 7 端的方案。

两个视角的 verdict：

| 视角 | 总体 | 核心问题 |
|------|-----|---------|
| Gemini 架构批判 | **must_fix × 4** / should_revise × 1 | 体积死锁、YAML 无解析主体、@HUMAN 伪防护、与开放标准兼容裂缝 |
| Claude 跨端落地 | **needs-revision** | Intent Ledger 5/7 客户端失效；动态 MCP 注入是未验证假设；@HUMAN 强制性只在 Claude Code 成立 |

---

## 2. 五大核心缺陷（必须修正）

### 🔴 缺陷 1：体积死锁（组件 1 × 组件 2 相互冲突）

> _"L2 <300 行" × "Intent Changelog 自动回填 AGENTS.md" = 文件单调膨胀 → AI 陷入'为记录意图而违反大小规则'的悖论。_

**修正**：Intent Ledger **物理隔离**到 `.intent-ledger.jsonl`，**禁止**回填到 `.md`。AGENTS.md 只保留由人手动筛选过的 "里程碑" 条目。

---

### 🔴 缺陷 2：YAML frontmatter 无原生解析主体

> _`scope / deps / priority` 没有任何客户端原生识别。只有 Fabric 自己的编译器 + MCP Server 能解析；非 MCP 客户端（当前 Copilot）会把 YAML 当噪声。_

**修正**：元数据语法降级为 **HTML 注释**：
```markdown
<!-- fab:scope=src/components/** -->
<!-- fab:deps=L0,L1/api -->
<!-- fab:priority=high -->
```
- 对未实现 Fabric 的工具链，这些行是纯注释 → **优雅降级，零污染**
- 对 Fabric 编译器，这是结构化指令
- 与 AGENTS.md 开放标准"无 schema 纯 MD"**兼容**，是扩展而非替代

---

### 🔴 缺陷 3：Hook 依赖让 Intent Ledger / @HUMAN 在 5/7 客户端失效

客户端 hook 支持真相：

| 客户端 | Hook |
|--------|------|
| Claude Code | ✅ 20+ |
| Windsurf | ✅ 12 |
| Gemini CLI | ✅ 11 |
| **Cursor / Copilot / Roo Code / Codex CLI** | ❌ **完全没有** |

**修正**：把"强制执行"的责任从**客户端 hook** 下沉到 **git hook（所有客户端共享）**
- Intent Ledger：**git pre-commit hook** 通过 `git diff` 反推当次变更，补写 `.intent-ledger.jsonl`（跨所有客户端一致，不挑 IDE）
- @HUMAN 保护：**git pre-commit hook** 检查 `@HUMAN` 包裹段是否被修改，若是则阻断提交
- 客户端 hook（Claude Code/Windsurf/Gemini CLI）作为**实时增强**，提前给 AI 友好提示；非实时版本走 git 兜底

---

### 🔴 缺陷 4：动态 MCP 注入是未经验证的假设

> _MCP 规范**没有标准的"会话中热更新"机制**——大部分客户端在会话启动时加载 MCP 配置，会话中不重新加载。"按活跃文件动态注入"在 Cursor/Windsurf/Roo Code 都**不能实现**。_

**修正**：把"动态注入"降级为 **"启动时路径感知静态注入"**
- Fabric MCP Server 在启动时根据 **工作目录 / 主要打开文件** 决定加载哪些 L1/L2 文件
- 放弃"生成代码过程中拦截式注入"的幻想
- Claude Code 独享实时 post-tool-use 拦截，作为可选强化

---

### 🔴 缺陷 5：与 AGENTS.md 开放标准的兼容裂缝

> _Fabric 强制 YAML frontmatter + 硬性嵌套结构 = **替代**了 AGENTS.md 的"无 schema 纯 MD"哲学；Docs-as-Views 编译产物（.cursorrules 等）等同于承认 AGENTS.md 普及失败，退回适配器地狱。_

**修正**：把 Fabric 明确定位为 **"AGENTS.md 的可选扩展层"**
- 不使用 Fabric 的读者/工具 → 看到纯 AGENTS.md（HTML 注释被忽略即可）
- 使用 Fabric 的读者/工具 → 解析 HTML 注释获得元数据
- 编译产物全部 `.gitignore`，只在本地和 CI 按需生成（避免产物入库的 merge 地狱）
- 顶部加 `# AUTO-GENERATED by fab compile — do not edit`

## 3. 修订后的 Fabric Lite（v1.1）蓝图

### 三层分离

```
┌─────────────────────────────────────────────────────┐
│ Layer A — 规范层 (committed to git)                  │
│   AGENTS.md (L0 根)                                   │
│   ├── <!-- fab:index -->                              │
│   ├── L1 领域层 (src/features/*/AGENTS.md, <300行)    │
│   └── L2 组件层 (src/components/*/AGENTS.md, <300行)  │
│                                                       │
│ Layer B — 意图层 (committed to git, append-only)      │
│   .intent-ledger.jsonl                                │
│                                                       │
│ Layer C — 分发层 (gitignored, locally generated)      │
│   .cursor/rules/*.mdc                                 │
│   .github/copilot-instructions.md                     │
│   .windsurf/rules/fabric.md                           │
│   CLAUDE.md (@AGENTS.md 导入)                         │
│   GEMINI.md (@import 链)                              │
└─────────────────────────────────────────────────────┘
```

### 四条强制规则（全端生效）

1. **规范层 A 是唯一真相源**；分发层 C 是派生产物，不入 git
2. **`<!-- fab:xxx -->` HTML 注释承载元数据**，任何工具链都能降级为"纯注释"
3. **git pre-commit hook** 做三件事：① 重编译 Layer C；② 补写 `.intent-ledger.jsonl`；③ 检查 `@HUMAN` 段完整性
4. **Layer A 的根 AGENTS.md 必含 `<!-- fab:index -->` 段**，列出所有 L1/L2 路径 + 一行描述，让 AI **一次决策**就能选择深入读哪些（解决 Gemini 指出的"树太深 token 低效"问题）

### 组件 × 客户端 × 可行性

| | L0/L1/L2 结构 | 元数据 HTML 注释 | Intent Ledger (git hook) | 静态 MCP 路径感知 | Docs-as-Views 编译 | @HUMAN 保护 (git hook) |
|---|---|---|---|---|---|---|
| Claude Code | ✅ | ✅ | ✅ + 客户端实时 | ✅ | ✅ | ✅ + 客户端 pre-tool-use 阻断 |
| Cursor | 🟡 编译 | ✅ | ✅ | ✅ | ✅ | ✅ |
| Windsurf | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ + 客户端 hook 提示 |
| GitHub Copilot | 🟡 编译 | ✅ | ✅ | 🚧 MCP 未 GA，用编译产物 | ✅ | ✅ |
| Roo Code | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| Codex CLI | ✅ | ✅ | ✅ | ❓ MCP 未确认，用 AGENTS.md 兜底 | ✅ (直接用原文) | ✅ |
| Gemini CLI | 🟡 编译 | ✅ | ✅ | ✅ | ✅ | ✅ + 客户端 hook 提示 |

**关键转变**：所有**强制性机制下沉到 git 层**（7 端一视同仁），客户端 hook 只做**实时增强**。

## 4. 第一周 MVP（只做最高 ROI 的两件事）

**Day 1-2 · 建立 Layer A + 基础 Fabric 规范**
- 项目根写 AGENTS.md（含 `<!-- fab:index -->` + 3-5 条 L0 宪法 + 若干 L1/L2 引用）
- 示范 1 个 L1 + 1 个 L2（验证 300 行约束）
- 写 `FABRIC.md`：定义 HTML 注释元数据语法 + `.intent-ledger.jsonl` schema

**Day 3-4 · 写 `fab` CLI（Node 脚本，单文件）**
- `fab compile` ：AGENTS.md → Layer C 分发产物（.cursorrules / CLAUDE.md / .windsurfrules / .github/copilot-instructions.md / GEMINI.md）
- `fab check` ：验证 L0/L1/L2 树大小 < 300 行、index 与真实文件一致
- `fab ledger append` ：从 git staged diff 生成一条 intent 记录
- 这三个命令 ≈ 400-600 行 JS

**Day 5-6 · git hook 集成**
- `.husky/pre-commit` 调 `fab compile && fab check && fab ledger append`
- （可选）Claude Code `settings.json` 加 post-tool-use hook 作为**实时增强**

**Day 7 · 5 端实测**
- Claude Code / Cursor / Windsurf / Copilot / Roo Code 各打开项目
- 给同一条 prompt，验证 L0 宪法被遵守、L2 规则在对应路径下被触发

**不在 MVP 范围**：
- ❌ 动态 MCP 实时注入（降级为静态路径感知，v2 再做）
- ❌ Codex CLI 深度集成（MCP 不确定）
- ❌ OS 只读权限层（引入部署复杂度，等概念验证后再议）

## 5. 三大 Kill Switch（Day 1-3 必须探测）

| Kill Switch | 探测方法 | 若触发 |
|------|---------|--------|
| **KS-1** `fab compile` 产物与源文件漂移 | Day 3 起 CI 加 `fab compile --check` | 必立即加 CI 守卫；产物文件顶加 `AUTO-GENERATED` 注释 |
| **KS-2** 团队 Cursor 占比 >40% 且无法接受"编译产物 .gitignore" | Day 1 团队调研 | 重新评估：要么允许产物入库+接受 conflict 成本，要么把 Fabric 定位为 CLI/非-Cursor-主力团队的方案 |
| **KS-3** Windsurf / Gemini CLI 的 post-tool-use hook 不能触发文件写 | Day 1-2 实测 | 实时 Intent Ledger 完全交给 git hook，不做客户端级实时记录（损失一点体验但不影响跨端一致性） |

## 6. 对比：原始 Fabric vs Fabric Lite（v1.1）

| 维度 | 原始 Fabric | Fabric Lite v1.1 |
|------|------------|------------------|
| 元数据格式 | YAML frontmatter（需 MCP 解析） | HTML 注释（所有工具降级兼容） |
| 强制性载体 | 客户端 hook | **git hook**（跨端一致） + 客户端 hook 作增强 |
| MCP 机制 | 动态注入（未经验证） | 启动时路径感知（已验证可行） |
| Intent Ledger | 实时 post-tool-use + 回填 .md | 物理隔离 .jsonl，git pre-commit 补录 |
| @HUMAN 保护 | Markdown 约定（无执行力） | git pre-commit 阻断 + AI System Prompt 强调 |
| 与 AGENTS.md 开放标准 | 替代（schema 冲突） | 扩展（注释级兼容） |
| 编译产物 | 默认入库 | `.gitignore`，本地/CI 生成 |
| 树深度 | L0→L1→L2 逐级 read_file | 根 index 预先列全路径，AI 一次决策 |
| 跨端覆盖 | 高宣称，实际 3/7 | 低调宣称，实际 7/7 底座 + 3/7 增强 |

## 7. 还需要用户确认的 3 个决策

1. **编译产物是否入库？** — 推荐 `.gitignore`（避免 merge 地狱，但开发者需 clone 后跑一次 `fab compile`）
2. **Intent Ledger 粒度？** — 每次 commit 一条 / 每次工具调用一条（后者仅在 Claude Code 有 hook 时可行）
3. **@HUMAN 保护强度？** — 仅 git hook 阻断（够用）/ 叠加 OS 只读（最强但增加配置成本）

## 8. Verdict

> **Fabric Lite v1.1 = go**（修订后）；**原始 Fabric = needs-revision**（如上 5 条修正必须做）。
> 核心思想保留（双层 + 意图 + 编译 + 治理 + 分级），但**强制力下沉到 git，元数据降级为 HTML 注释**。
