# Fabric v1.0 首发故事脚本

Soul Reference

主标语 / Main Tagline

人机协作的语义共识平面

The Consensus Plane for AI-Human Collaboration

命名约定 / Naming

`fab` = command

`Fabric` = product

`fabric` = UI wordmark

版本约定 / Versioning

Fabric v1.0 是发布起点。

Dashboard 明确归入 v1.1 体验。

三面分工 / Three Planes

CLI = control plane

Dashboard = observability plane

MCP = runtime dispatch

语言策略 / Language Policy

默认 locale 是 `en`。

英文硬规则保留 `MUST`、`NEVER`、`fab_get_rules`、MCP tool 名、JSON key、file path、class name 这类 protected token。

中文解释层服务人类维护者，不稀释 AI 的执行约束。

npm scope

统一使用 `@fabric/*`。

叙事视角

以下全文都站在“仓库维护者 / AI Enablement Owner”的第一人称视角来写。

我不是来围观一个炫技工具。

我是来把一支已经在交付的团队，重新放回一个可验证、可维护、可扩展的协作秩序里。

## 前言：谁是 Reference User

我是一个 6 人 Cocos Creator 游戏团队的仓库维护者。

我们在做一个狼人杀小项目。

项目不大。

但协作面已经够复杂。

程序、美术技术支持、玩法策划和我自己，都在不同时间段借助 AI 写代码、补注释、改配置、修文档。

团队里没有人只用一个客户端。

有人偏爱 Claude Code。

有人在 Cursor 里迭代得更快。

有人在 Codex 里做局部实现。

也有人临时在 Windsurf、Roo、Gemini 里做对照验证。

这些客户端都很强。

但它们不会天然共享我的项目边界。

我的真实痛点不是“AI 不会写代码”。

而是“不同 AI 会把同一个仓库理解成不同的东西”。

我今天刚在 Claude Code 里强调过：

`@ccclass` decorator 不能乱动。

到了晚上，另一个同事在 Cursor 里让 AI 重构 `Player.ts`，又把 decorator 名称和脚本绑定关系改掉了。

我昨天刚把 `assets/**/*.meta` 列成禁区。

第二天就有人在 Codex 里批量整理资源目录时，顺手把 `.meta` 一起改了。

我在评审里一次次解释：

狼人角色机制是策略核心。

夜晚结算顺序不是 UI 细节。

角色配置不是“看起来能抽成 JSON 的普通对象”。

但是只靠人肉守门，规则会漂。

只靠 PR 评论，记忆会丢。

只靠我口头重复，团队会疲劳。

最后的结果，不是某一次大事故。

而是 commit 质量越来越不均匀。

有的提交结构清楚，能追溯意图。

有的提交只是“AI 觉得这样更整洁”。

有的提交表面通过编译，但把项目真正不能动的东西碰了。

这正是我需要 Fabric 的原因。

我需要一个不是靠情绪、不是靠会议纪要、也不是靠某一个 AI 会话上下文生效的规则面。

我需要一块真正能被织进去、能被分发、也能被看见的语义共识平面。

对我来说，Fabric v1.0 的意义，不是“又一个 AI 插件”。

它是我第一次能用一个明确的安装动作，把规则写进仓库边界里。

`fab init` 负责落规。

MCP 负责分发。

Dashboard 在 v1.1 里负责把这个规则生态展开给我看。

这也是为什么这份脚本必须站在我的视角写。

我不是抽象用户。

我是那个要对 release 质量负责的人。

我是那个要解释“为什么这个 commit 该挡住”的人。

我是那个要让 Claude Code、Cursor、Codex 这些工具在一个项目里共存的人。

我也是那个最终要回答一句话的人：

“当团队用了很多 AI 之后，仓库规则到底是谁在维护？”

在 Fabric 的叙事里，这个答案不再是“我一个人硬扛”。

答案变成：

我把规则织进仓库。

然后让每一个客户端都在同一张 fabric 上工作。

狼人杀这个项目，正好是一个足够真实的样本。

它有明确框架。

它有非常具体的玩法边界。

它也有典型的 AI 漂移风险。

`Game.ts` 看起来很小。

`Player.ts` 看起来很通用。

`Network.ts` 看起来像个占位。

但只要维护者真正碰过这种项目，就知道最危险的地方从来不是代码长短。

而是“语义被误判”。

角色机制一旦被抽象错。

网络边界一旦被模糊。

prefab、scene、`.meta` 一旦被误改。

项目就会从一个还能继续迭代的仓库，变成一个表面能跑、内部已经失真、只能靠人慢慢捡回来的仓库。

所以我把 Reference User 定义得很具体。

我是一个维护者。

我是一个 AI Enablement Owner。

我不追求神奇。

我追求的是：

规则先落地。

解释可复述。

边界可验证。

团队可以持续使用，不需要每周重新宣讲一遍。

如果 Fabric v1.0 的首发故事成立，它就应该让我在第一个上午完成初始化。

让我在第一天就把规则写成真实文件。

让我在第二天开始看到规则生态的状态。

这就是下面三幕的意义。

它们不是概念图。

它们是我真正会拿来给团队看、也能直接截进白皮书的用户体验片段。

## 第一幕：Day 1 上午，`fab init` 中文化交互流

时间是周一上午九点二十。

我把狼人杀项目切到一个干净分支。

这个仓库里还没有 `AGENTS.md`。

也还没有 `.fabric/`。

我做的第一件事，不是开浏览器。

也不是写一份新的规范文档。

我直接在项目根目录运行：

```bash
fab init
```

我期待的不是一段“安装成功”的空泛反馈。

我期待 Fabric 用一次很短的 CLI 交互，把三件事同时做对：

第一，识别它面对的是一个什么项目。

第二，把最小可用的协议骨架写进仓库。

第三，明确告诉我下一步该怎么把语义初始化接到 AI 客户端里。

下面这段 stdout，是我希望在白皮书里原样出现的首发版本中文化样例。

```text
$ fab init

这一屏里，真正重要的不是“Created”这几个字。

真正重要的是，我第一次看到 CLI 把“项目被识别成什么”和“规则接下来如何继续生成”放在同一条体验链路上。

它不是只生成一个文件就结束。

它也不是把所有语义判断硬塞进 CLI。

它先快、先准、先非破坏性地落盘。

然后把需要 AI 参与的部分，显式 handoff 给下一步。

这一点对维护者非常关键。

因为我最怕的并不是 AI 参与。

我最怕的是 AI 在一个看不见契约的黑箱里参与。

Fabric 在这里做的事非常具体：

它让 `.fabric/forensic.json` 先成为证据包。

它让 `.claude/skills/agents-md-init/SKILL.md` 成为语义初始化入口。

它让 `.claude/hooks/agents-md-init-reminder.cjs` 和 `.claude/settings.json` 成为跨会话兜底。

它还用一条 `Reason:` 明确说明：

这里有一部分信息必须保持英文。

不是因为产品没本地化。

而是因为 prompt contract 需要 protected token 保持稳定。

这就是我想要的 i18n 分层。

人类面对 CLI 运行时输出时，看到中文。

AI 面对硬规则和调用触发词时，拿到英文 token。

两层互不打架。

也不互相污染。

我执行完 `fab init` 之后，没有再回到终端里猜下一步。

我直接打开 Claude Code。

此时我不是在“求 AI 帮忙写一份规范”。

我是在继续一个已经被 CLI 明确准备好的初始化事务。

我在 Claude Code 里输入的第一句话，可以非常普通：

```text
我刚执行了 fab init，请使用 agents-md-init 完成当前项目初始化。
```

随后，`agents-md-init` 的 3-Phase 访谈开始。

这段访谈的价值，不在于“多聊几轮”。

而在于它把应该由维护者确认的东西，精确地留给维护者确认。

把应该由证据包回答的东西，交给证据包。

把应该由生成器自动落地的东西，交给生成器。

下面是我希望文档里直接引用的 Phase 1 到 Phase 3 脚本。

### Phase 1：框架确认 / Framework Confirm

```text
AI: 我先确认 framework assumptions。
AI: 我检测到这是一个 Cocos Creator 3.8 项目，主要脚本位于 `assets/scripts`，当前可见 `Game.ts`、`Player.ts`、`Network.ts` 三个 TypeScript Component。
AI: 我看到三个共同特征：
AI: 1. class 都继承 `Component`
AI: 2. 都使用 `@ccclass("<ClassName>")`
AI: 3. framework evidence 来自 `project.config.json` 与脚本装饰器
AI: 请确认：你的项目是 Cocos Creator 3.8 吗？

我: 是，版本就是 3.8。

AI: 请再确认一件事：这些脚本是 TypeScript 主线，不会退回 JavaScript，对吗？

我: 对，TypeScript 是主线。
```

这一步很短。

Fabric 不在这里问“你想要什么代码风格”。

它只确认事实层。

如果第一层事实没锁住，后面所有规则都会虚。

### Phase 2：不变式提取 / Invariants Extraction

确认完框架之后，AI 开始问真正该问的硬边界。

它的问题是中文。

但关键技术术语，比如 `decorator`、`file path`、`class name`、`update()` 这种不会被翻掉。

```text
AI: 下面我只提取 invariants，不讨论 preferences。
AI: 请按 yes / no / 具体规则回答。

AI: 1. `@ccclass("Game")`、`@ccclass("Player")`、`@ccclass("Network")` 这些 decorator 绑定，是否视为受保护的 script identity，不允许 AI 擅自改名或删除？

我: 是，script identity 要保护。

AI: 2. 所有 `**/*.meta` 文件是否都不可修改？

我: 是，`.meta` 一律不改。

AI: 3. `assets/prefabs/**` 与 `assets/scenes/**` 是否视为 protect surface，默认禁止 AI 直接改动？

我: 是，这两个目录都要保护。

AI: 4. `update()` 是否禁止使用 `async`？

我: 是，`update()` 不能 async，`lateUpdate()` 也不能。

AI: 5. 狼人杀角色机制是否属于策略核心，任何涉及 role distribution、night resolution、win condition 的改动都必须先显式说明影响？

我: 是，这条一定要写进解释层。
```

这时我已经能感觉到区别了。

以前我在 AI 会话里说规则，常常像在留言板上写便签。

这次不是。

这次每一个回答都在被组织成可落盘的约束。

它们不是对当前聊天窗口负责。

它们是对仓库负责。

### Phase 3：构造落地 / Domain Construction

到了第三阶段，AI 不再问泛泛的问题。

它开始把证据包和我的回答，压成实际结构。

这里最关键的一问，是 domain 如何组织。

```text
AI: 我准备生成分层 `AGENTS.md`。
AI: 当前脚本只有三个核心文件：
AI: - `assets/scripts/Game.ts`
AI: - `assets/scripts/Player.ts`
AI: - `assets/scripts/Network.ts`
AI: 我建议把它们拆成两个 domain：
AI: - `gameplay`: `Game.ts` + `Player.ts`
AI: - `network`: `Network.ts`
AI: 请确认：分成 `gameplay` / `network` 两个 domain，还是先合成一个 domain？

我: 分成两个 domain。狼人杀玩法和网络边界要分开描述。

AI: 明白。我会生成：
AI: - `.fabric/init-context.json`
AI: - 完整 `AGENTS.md`
AI: - 更新后的 `.fabric/agents.meta.json`
AI: 规则层保持英文硬约束，最后追加中文解释层，专门说明 werewolf 机制不可破坏。
```

几秒之后，Claude Code 给出最后确认。

```text
已生成 `AGENTS.md`，共 180 行。
已生成 `.fabric/init-context.json`。
已更新 `.fabric/agents.meta.json` revision。

建议下一步：
1. 运行 `fab hooks install`
2. 首次提交前执行 `fab sync-meta`
3. 如需观测规则状态，在 v1.1 使用 `fab serve`
```

对外讲述时，这一幕最重要的感受不是“中文化做得不错”。

而是维护者终于第一次得到一个完整且可重复的初始化闭环：

CLI 先把协议织进去。

AI 再把语义补齐。

整个过程里，谁负责什么，非常清楚。

`fab` 不是生成器大杂烩。

它是 control plane。

Claude Code 在这一幕里也不是“万能助手”。

它是接过 evidence 和 hard rules 之后，完成语义初始化的执行者。

这就是我愿意把它拿给团队看的原因。

每个人都能复述。

每一步都能验证。

每个 artifact 都在磁盘上。

它保留 Fabric v1.0 的 header。

并在最后一段提供中文解释层，告诉 AI 这个项目最不能误判的地方是什么。

```md
# werewolf-minigame-stub / Fabric v1.0

> 人机协作的语义共识平面 / The Consensus Plane for AI-Human Collaboration
>

## Repository Identity

- Project type: Cocos Creator 3.8 TypeScript component project
- Primary persona: repository maintainer / AI enablement owner
- Runtime language for humans: zh-CN first-class
- Protected token layer for AI: English hard rules only
- Package scope convention: `@fabric/*`

## Scope Map

- Root contract:
  - `AGENTS.md`
- Gameplay domain:
  - `assets/scripts/Game.ts`
  - `assets/scripts/Player.ts`
- Network domain:
  - `assets/scripts/Network.ts`
- Protected assets:
  - `assets/prefabs/**`
  - `assets/scenes/**`
  - `**/*.meta`

## Hard Rules

### MUST

- MUST call `fab_get_rules(path=<file>)` before editing any file when MCP is available.
- MUST preserve `@ccclass("Game")`, `@ccclass("Player")`, and `@ccclass("Network")` unless a human explicitly approves a rename.
- MUST keep `Game`, `Player`, and `Network` as `Component` subclasses.
- MUST keep imports aligned with Cocos Creator 3.8 patterns and import engine symbols from `cc`.
- MUST keep `update()` and `lateUpdate()` synchronous.
- MUST treat `.meta` files as immutable.
- MUST treat `assets/prefabs/**` and `assets/scenes/**` as protected surfaces.
- MUST explain gameplay impact before changing role distribution, night resolution, or win condition logic.
- MUST keep changes small, reversible, and easy to diff.
- MUST update `AGENTS.md` and `.fabric/agents.meta.json` together when repository rules change.

### NEVER

- NEVER remove or rename an existing `@ccclass` decorator string without human approval.
- NEVER edit any `**/*.meta` file.
- NEVER edit `assets/prefabs/**` or `assets/scenes/**` unless the task explicitly grants approval.
- NEVER introduce `async` into `update()` or `lateUpdate()`.
- NEVER move gameplay authority into `Network.ts`.
- NEVER hide werewolf role mechanics behind generic abstractions that erase day/night semantics.
- NEVER rewrite a human-locked region.
- NEVER change file ownership boundaries between `gameplay` and `network` silently.
- NEVER replace concrete game terms with vague placeholders such as `unit`, `entity`, or `session actor` when the code is about werewolf roles.

## Working Contract

- Human-readable explanations can be Chinese.
- Protected constraint lines stay in English.
- If human language and hard rule layer conflict, the hard rule layer wins until a maintainer edits this file.
- If code and this file disagree, pause and ask a human before proceeding.

## Framework Invariants

### Engine

- Cocos Creator version is `3.8.0`.
- TypeScript is the primary authoring language.
- Script identity is bound by file path + class name + `@ccclass("<ClassName>")`.
- Component lifecycle assumptions must stay compatible with synchronous `start()`, `update()`, and scene-driven mounting.

### File Safety

- Protect `**/*.meta`.
- Protect `assets/prefabs/**`.
- Protect `assets/scenes/**`.
- Protect any future file containing `@HUMAN START` and `@HUMAN END` markers.

### Async Boundaries

- `update()` MUST remain synchronous.
- `lateUpdate()` MUST remain synchronous.
- If async behavior is required, isolate it outside frame-step functions and explain the scheduling impact.

## Domain Index

- `gameplay`
  - Owns match lifecycle, role state, round progression, and win conditions.
- `network`
  - Owns transport, message serialization, and remote sync boundaries.

## Domain: gameplay

### Owned Files

- `assets/scripts/Game.ts`
- `assets/scripts/Player.ts`

### Purpose

- `gameplay` owns the game model that a player experiences.
- `gameplay` is where role assignment, night/day flow, and victory judgement belong.
- `gameplay` may depend on networking signals, but it must not let transport structure redefine game rules.

### Gameplay MUST

- MUST keep `Game.ts` as the orchestration point for match phases.
- MUST keep `Player.ts` focused on player-facing state and per-role status.
- MUST describe any change that affects role balance, elimination order, or information asymmetry.
- MUST keep naming concrete to the werewolf game model.
- MUST prefer explicit role terminology such as `wolf`, `villager`, `seer`, `witch`, `guard`, `night`, and `day` when those concepts exist.

### Gameplay NEVER

- NEVER flatten all roles into a generic `type` system without preserving their distinct win and action semantics.
- NEVER merge phase logic into a single opaque resolver if that makes day/night ordering untraceable.
- NEVER make `Player.ts` a dumping ground for network transport state.
- NEVER encode strategy-critical logic only in comments.

### File Notes: `assets/scripts/Game.ts`

- `Game.ts` is the top-level gameplay coordinator.
- It should remain the first place a maintainer reads to understand phase progression.
- Safe changes:
  - clarify method names
  - extract tiny pure helpers
  - add guard rails for invalid transitions
- Unsafe changes without explicit explanation:
  - rewriting role assignment flow
  - changing night resolution order
  - changing win condition precedence
  - moving orchestration authority into another file

### File Notes: `assets/scripts/Player.ts`

- `Player.ts` represents player-facing runtime state.
- It should keep state names semantically tied to werewolf gameplay.
- Safe changes:
  - add clear state fields
  - strengthen validation around state transitions
  - clarify per-role capability boundaries
- Unsafe changes without explicit explanation:
  - converting role semantics into a generic attribute bag
  - hiding alive/dead/public/private state behind ambiguous booleans
  - renaming class identity or decorator strings

## Domain: network

### Owned Files

- `assets/scripts/Network.ts`

### Purpose

- `network` owns transport and synchronization.
- `network` is a boundary, not the source of gameplay truth.
- `network` may carry role or phase data, but it must not redefine role rules.

### Network MUST

- MUST keep `Network.ts` as the entry point for transport logic in this stub.
- MUST keep protocol changes explicit and reviewable.
- MUST annotate any data contract change that can alter gameplay interpretation.
- MUST keep serialization and dispatch concerns separable from gameplay resolution.

### Network NEVER

- NEVER move core win-condition logic into `Network.ts`.
- NEVER let network retry or reconnect logic mutate gameplay truth silently.
- NEVER rename message fields in a way that obscures domain meaning without a migration note.
- NEVER let `Network.ts` become the only place where phase ordering is understood.

### File Notes: `assets/scripts/Network.ts`

- `Network.ts` may grow, but its responsibility stays narrow.
- Safe changes:
  - add message wrappers
  - add validation
  - add logging around transport boundaries
- Unsafe changes without explicit explanation:
  - encoding role resolution directly in transport handlers
  - mutating `Player` state without a gameplay boundary call
  - introducing hidden side effects during connect/disconnect

## Human Lock Policy

- Human locks protect strategy-critical text and configuration.
- AI may read human-locked regions.
- AI must not edit human-locked regions unless a maintainer explicitly unlocks them.
- If a task depends on changing a locked region, stop and ask for approval.

## `@HUMAN` Lock Range Example

```ts
// file: assets/scripts/Game.ts

// @HUMAN START role-balance-config
const ROLE_SETUP = {
  wolf: 2,
  villager: 4,
  seer: 1,
  witch: 1,
};
// @HUMAN END role-balance-config
```

- The example above shows the kind of region that belongs to a maintainer.
- Even if the exact block is not present yet, Fabric should treat future role-balance ranges as protected by default.

## Drift Expectations

- If `AGENTS.md` changes, run `fab sync-meta`.
- If a human lock hash changes, run the approval flow before normal development resumes.
- If MCP is connected, use `fab_append_intent` to preserve intent traceability for large changes.

## Review Checklist

- Did the change preserve `@ccclass` identities?
- Did the change avoid `**/*.meta` edits?
- Did the change keep `update()` synchronous?
- Did the change keep gameplay and network responsibilities separate?
- Did the change explain impact on werewolf role semantics?
- Did the change avoid touching protected asset directories?

## 第三幕：Dashboard 初见时刻

这件事发生在第二天早上。

准确地说，是 Fabric v1.1 的体验。

前一天我已经完成了 v1.0 的初始化和 hooks 安装。

仓库里已经有了：

`AGENTS.md`

`.fabric/agents.meta.json`

`.fabric/human-lock.json`

`.fabric/init-context.json`

还有开始积累的 `.intent-ledger.jsonl`

这时候，Dashboard 的角色就清楚了。

它不是第二个编辑器。

它也不是一个另起炉灶的配置中心。

它是 observability plane。

它把前一天已经织进去的规则，摊开给我看。

我运行：

```bash
fab serve
```

然后浏览器自动打开：

```text
http://127.0.0.1:7373
```

第一页出现时，我看到的不是一块空白管理后台。

而是一块已经接上仓库状态的工作台。

我希望首发脚本里把这第一眼写得足够具体。

不是为了漂亮。

是为了让维护者一眼明白：

这里看到的每一块信息，都对应仓库里已经存在的真实 artifact。

### 首屏第一眼

浏览器刚打开时，顶部 header 先抓住我。

左上角是小写的 `fabric` wordmark。

旁边有一个方框里的 `F` lettermark。

右侧挂着 `v1.1.0` 徽章。

再往右，是一个亮起的 `CONNECTED` 状态。

这个状态不是装饰。

它告诉我 SSE 和本地服务已经连通。

header 下方不是空 slogan。

它用一行很短的副文本把产品位置讲清楚：

人机协作的语义共识平面

这时我已经能把 CLI 的第一幕和当前页面接起来了。

前一天 CLI 负责把规则织进去。

现在 Dashboard 负责把这些 thread 组织成我能观察的结构。

### 左侧导航

左侧导航不是全英文，也不是只给中文昵称。

它采用中文主 / 英文副的并排方式。

我看到的顺序是：

- 规则树 Rules Tree
- 人类仓库 Human Lock
- 意图时间线 Intent Timeline
- 诊断 Doctor
- 历史回放 History Replay

这套命名很重要。

中文主标签让我团队里所有人第一次看就知道每块是干什么的。

英文副标签又把术语和 API、文档、MCP tool 里的命名接起来。

它不会让我在沟通时陷入“页面上一个叫法，代码里另一个叫法”的裂缝。

### 默认页：规则树 Rules Tree

默认打开的是规则树 Rules Tree。

这很对。

因为维护者第一次进来时，最想确认的不是“最近谁点了什么按钮”。

而是“当前仓库规则到底长什么样”。

我看到的主体内容可以用下面这张近似线框来描述：

```text
fabric [F]  v1.1.0                                    CONNECTED
人机协作的语义共识平面

规则树 Rules Tree

这里每个节点都不是装饰性的树节点。

这给我的感受很直接。

以前 `AGENTS.md` 像一份静态文件。

现在它第一次像一棵可以被检查、被比较、被追踪 revision 的结构树。

对维护者来说，这就是可管理性的开始。

### 人类仓库 Human Lock

我点进左侧第二项：

人类仓库 Human Lock

这一页最重要的不是数量。

而是“哪些东西被人类明确声明为不能漂”。

在狼人杀这个样例里，我只需要一个锁就够了。

因为好的首发故事不需要堆太多案例。

它只需要一个足够说明问题的锁。

我希望页面里直接出现这样一张卡片：

```text
Human Lock #1

range: 18-29
label: role-balance-config
status: no drift

状态是绿色。

页面右侧给出一个简单但明确的结论：

无漂移

这四个字对维护者很重要。

团队已经开始把真正关键的游戏平衡区域，和普通实现区域区分开。

### 意图时间线 Intent Timeline

接下来我会看意图时间线 Intent Timeline。

这里最打动我的不是“有日志”。

而是日志第一次按人类和 AI 的双源语义被排在一起。

Fabric 不再假装所有 commit 都来自同一个抽象作者。

它承认：

这个仓库里本来就有两种来源。

而维护工作恰恰是理解这两种来源如何交替塑形。

我希望首屏就看到最近 5 条记录。

AI source 用 indigo 点。

Human source 用 teal 点。

两种颜色交替出现。

每条都带一个足够具体的 diff 摘要。

| Source | Time | Diff Summary |

| Human | 08:43 | approve lock hash after balancing `ROLE_SETUP` comments |
| AI | 08:31 | `Network.ts`: add explicit transport validation before dispatch |

这五条记录放在一起的时候，页面给我的信息密度远高于“最近五个 commit”。

Intent Timeline 告诉我：

是谁先动的。

为什么动。

后续有没有人类仲裁。

改动大概落在哪个语义边界上。

这是我第一次在一个屏幕上看到“规则生态”而不是“文件列表”。

### 第一眼的心理落点

当这一屏真的完整出现的时候，我脑子里只会冒出一句话：

“我第一次在一个屏幕看到整个规则生态的状态。”

不是整个项目状态。

也不是整个开发状态。

而是整个规则生态。

这句话很关键。

因为 Fabric 讲的从来不是 IDE 替代。

它讲的是共识被织进去之后，如何被维护者持续看见。

### Daily Loop：团队进入日常节奏之后

真正证明 Dashboard 不是演示页面的，不是首屏。

而是它进入 daily loop 之后会不会继续有用。

我想象一个很普通的上午十点十分。

团队里的同事在 Cursor 里完成了一次小改动。

他提交前，pre-commit 三件套跑过：

`fab sync-meta --check-only`

`fab human-lint`

`fab ledger-append --staged`

如果一切正常，commit 成功。

这时 Dashboard 不需要我手动刷新。

浏览器右上角的 `CONNECTED` 还亮着。

SSE 会主动推来一条新事件。

我在意图时间线页里，会直接看到一条新记录插到最上面。

这个瞬间，我希望脚本里具体写成这样：

```text
event: ledger:appended
source: ai
summary: `Game.ts`: clarify vote-resolution branch after night phase
ts: 2026-04-20 10:10:42
```

如果这次提交没有碰任何受保护区域。

Human Lock 页继续保持绿色。

如果这次提交碰到了 `@HUMAN` 锁定范围，或者改动导致锁 hash 漂移。

Human Lock 卡片会从绿色变成橙色或红色。

Doctor 页的总体状态也会跟着跳成 `warn` 或 `error`。

我不需要等某个人来群里解释：

“刚才那个改动可能有点问题。”

因为 observability plane 已经先把问题摆出来了。

这时我的工作不再是四处追问。

我的工作变成：

看见 drift。

判断 drift 是合理演进，还是违规偏移。

如果是合理演进，我批准新的 hash。

如果不是，我在意图时间线里就能追到是谁、为什么、从哪个客户端来的。

这就是 Dashboard 在 v1.1 应该成立的价值。

它不制造新规则。

它把已经存在的规则、锁和意图，变成可见的维护界面。

对我这个维护者来说，第二天早上的体验不只是“有个新页面可以看”。

而是整个产品叙事第一次闭合了。

第一天，CLI 把规则织进项目。

第二天，Dashboard 把规则展开给我看。

MCP 始终在背后做 runtime dispatch。

三面终于不是文档里的抽象框图。

而是一套连续的动作。

## 后记：从这里到哪里

如果我要用一句最短的话概括这份脚本证明了什么，

那就是：

Fabric v1.0 已经足以交付“CLI 织入规则，Dashboard 展开规则”的产品叙事。

v1.0 不需要假装自己已经把所有观测能力都做完。

它只需要把第一步做扎实。

也就是让维护者能在一个上午完成：

安装。

识别。

落规。

AI handoff。

然后把这份规则真正留在仓库里。

当这一步成立，v1.1 的方向就很自然。

它不是另起主题。

它是 Observable Maintenance。

它回答的是：

规则既然已经被织进去了，

维护者要如何持续看见它们、审核它们、追踪它们。

再往后，v1.2 的方向也清楚。

它是 Portability & Trust。

它回答的是：

这套规则如何跨项目迁移。

如何在更多客户端之间保持一致。

如何让团队对 drift、迁移、审计和信任链形成更稳定的把握。

对我这个仓库维护者来说，

这条路线不是从功能列表推出来的。

它是从每天真实要面对的问题推出来的。

我需要先把规则织进去。

然后把它们看见。

再让它们可以被带走、被验证、被信任。

这就是 Fabric 从 v1.0 到 v1.2 最可信的一条演进线。

也是这份 Soul Reference 想留下的核心感受：

不是某个 AI 更聪明了。

而是整个仓库第一次拥有了一张可落地、可分发、可观测的语义共识平面。
